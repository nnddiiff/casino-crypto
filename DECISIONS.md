## Рамка сдачи (под кого оптимизируем)
- Это **frontend-assessment для iGaming-компании** → оценивать будут продуктово, не только инженерно. Линза iGaming-фундамента: воронка Visit→Registration→FTD (first-time depositor)→Retention; юнит-экономика (house edge, hold %, RTP, GGR/NGR/LTV); принцип «AI должен двигать метрики, а не быть демо-фичей ради AI».
- Следствия для сборки:
  1. **Онбординг = снижение трения воронки.** Путь от входа до первой ставки — это Registration→FTD; держим безгазовый вход и быстрый первый бросок, «много счетов» убираем (убийца конверсии). См. открытый вопрос «Ревизия UX».
  2. **Дашборд самоаудита = язык юнит-экономики** (edge/hold/RTP) + проверяемая честность = доверие игрока (retention). Сильнейший артефакт «понимаю бизнес казино, а не просто накодил игру».
  3. **README/Loom — на языке продукта/метрик**; глубоко объяснить провабли-фейр (доверие), честно — самое сложное неизвестное.
  4. **AI-бонус — честно про результат и где AI подвёл; без «AI ради AI» и фейковых AI-фич.**
- Зрелость для такого оценщика = продуктовое мышление (воронка+юнит-экономика) + инженерная глубина (честность/безопасность) + чистый вылизанный рабочий продукт + честный AI, показанные его языком. Это рамка всей ревизии и сдачи.

## Модель счёта и кошелька (ФИНАЛ — зафиксировано)
Для пользователя — ровно две сущности, как в любом казино:
- **Счёт казино** — один баланс, с которого играешь. Технически это `balances[player]` в контракте. Это НЕ лишняя сущность: для провабли-фейр казино с асинхронным рандомом (Pyth запрос→колбэк) средства ОБЯЗАНЫ лежать в контракте между двумя транзакциями — иначе колбэк не выплатит/не откатит. Подтверждено разведкой по индустрии: это доминирующий паттерн on-chain казино (deposit→balance→bet→withdraw); и бриф прямо требует «внести в казино» + «вывести обратно».
- **Личный кошелёк** — независимый внешний кошелёк игрока (MetaMask/биржа). НЕ «привязывается» к аккаунту (это как привязать чужой банковский счёт — невозможно по определению). Это просто **адрес назначения выплаты** при выводе, как IBAN. Так и на Stake/Rollbit: вводишь адрес вывода.

Поток: вход по email → **Счёт казино** (один баланс) → играешь → вывод на личный адрес.

Что прячем (плумбинг, пользователь не видит как отдельные сущности):
- Встроенный кошелёк (email + AA) — это и ЕСТЬ on-chain форма «Счёта казино», а не отдельный «твой кошелёк». Слово «кошелёк» к нему в UI не применять.
- Разделение встроенный-кошелёк-ETH ↔ `balances[]` — скрываем автодепозитом: на «Сыграть», если в контракте не хватает, фронт незаметно делает `deposit()` нужной суммы (gasless, без подтверждений). Кран тоже автоматически в игровой баланс.
- EOA-админ AA — невидим всегда.

Решения, вытекающие отсюда:
- **Редеплой контракта НЕ нужен** (Вариант C отклонён). `balances[]`-модель — правильная и стандартная, оставляем. Комиссию Pyth (`msg.value` из встроенного кошелька) покрываем фронт-буфером, пользователю невидимо.
- **Ручную отправку на адрес НЕ убираем как концепт, а переосмысляем**: это и есть «Вывести на свой кошелёк» (ввод адреса назначения) — единственный и легитимный вывод. Под капотом: снять из контракта → отправить на адрес, в один клик, gasless. Лишним был не сам ввод адреса, а то, что он торчал третьей непонятной сущностью рядом со встроенным кошельком.
- Всё это — на фронте, без изменения контракта.

## Стек и сеть
- Сеть: **Base Sepolia** — EVM (надёжная AI-кодогенерация + экономия токенов), быстро/дёшево, лучший paymaster+AA, есть Pyth Entropy, считается «Ethereum testnet» (L2). В README объяснить выбор явно.
- Деньги: **нативный тестовый ETH + кран в контракте** — нет деплоя токена, нет approve, простейшая петля deposit/withdraw.
- Случайность: **Pyth Entropy** — проверяемая on-chain, дешевле/быстрее Chainlink VRF, задеплоена на Base Sepolia, есть готовый пример под Foundry. Откат: commit-reveal.
- Инструменты: **Foundry** (контракт: write/test/deploy/verify) + **thirdweb v5** (фронт, кошелёк, AA, sponsorGas).
- Фронт-фреймворк: **Next.js 16 (App Router) + TypeScript**, src-dir, alias `@/*`. Почему: Vercel-native деплой без конфига, shadcn/ui и thirdweb максимально документированы под Next → меньше итераций AI; кошельковые хуки client-only закрываются провайдером-обёрткой. Внимание: это Next **16** (не 15) — ломает часть привычных API, сверяться с `web/node_modules/next/dist/docs/` (предупреждение в `web/AGENTS.md`).
- Закреплённые версии каркаса (воспроизводимость): solc 0.8.28 (evm cancun, optimizer 200), forge-std 1.16.1, OpenZeppelin 5.6.1, thirdweb 5.120, Tailwind v4, shadcn style `base-nova`.
- Сборка контракта: примитивы из **OpenZeppelin** (`ReentrancyGuard`, `Ownable`), интеграцию Pyth — из официального `coin_flip`-примера; логику Limbo (выплата, проверка, edge, банк/кран) **пишем сами**. Целый казино-контракт не форкаем — унаследованные баги с деньгами + теряется понимание для Loom/«hardest unknown».
- Онбординг: thirdweb account abstraction + безгазовый вход — **реализован** (`accountAbstraction={{ chain, sponsorGas: true }}` на `ConnectButton`: оборачивает и встроенный кошелёк, и MetaMask в смарт-аккаунт ERC-4337, газ спонсируется бандлером/пэймастером thirdweb; на тестнете бесплатно, без биллинга). Безгазовая петля **проверена on-chain** (смарт-аккаунт `0xc4911d48…`: кран→депозит→ставка→вывод→отправка на внешний адрес, газ спонсирован, 0 обычных tx). ⚠️ Но безгазовый встроенный кошелёк породил **UX-путаницу**: лишние уровни счёта (email-смарт-аккаунт → баланс в игре → внешний адрес) и ручной ввод внешнего адреса. Это главная причина «не готов к выпуску» — см. открытый вопрос «Ревизия UX». Также проп оборачивает ВСЕ кошельки модалки, включая MetaMask (адрес становится смарт-аккаунтом). Причина не откатывать на чистый MetaMask: газ Base Sepolia не достать (краны платные/нерабочие, см. память [[base-sepolia-gas-hard-use-gasless]]).
- Дизайн/UI: **Claude Code + shadcn/ui** (Tailwind) под конкретный референс (Stake/Shuffle/BC.Game) — один инструмент, бесплатно; v0.dev не берём (скудный free-лимит + второй тул, при референсе качество то же). Фронт/полировка — приоритет; «вау» = анимированная центральная цифра множителя + тёмная тема с ярким акцентом + провабли-фейр-бейдж + mobile-first.

## Игра
- **Limbo** — игрок задаёт целевой множитель, генерируется случайный, выигрыш если ≥ цели. Один розыгрыш, простейший контракт, edge прозрачен (вероятность = RTP/T). Реальная игра (Hacksaw/Stake).

## Отличие (дифференциатор)
- **Дашборд самоаудита**: live edge/RTP/hold + ссылка на пруф случайности к каждой ставке + факт банка казино против теории. Это и есть «не скам» + коммерческая грамотность.
- **Верификатор честности (реализован)**: отдельная панель по seq ставки тянет сырое `randomNumber` из события Pyth `Revealed` (V2-форма, сверена on-chain; контракт `0x41c9…0d4c`), пересчитывает множитель публичной `previewMultiplier` контракта и показывает совпадение с записанным `BetSettled`. Закрывает находку аудита slot 2 (`BetSettled` не публикует `randomNumber`) без редеплоя. Поиск без индексатора (Insight у thirdweb домен-ограничен → `useIndexer:false`): для ставки сессии — по точному блоку колбэка, иначе якорь по `bets.requestedAt` + окна ≤1000 блоков (лимит RPC `eth_getLogs`). Видна без кошелька.

## Реализационные инварианты (на стороне ассистента, не груз пользователя)

- Кран: защита от reentrancy (checks-effects-interactions / ReentrancyGuard), один claim на адрес.
- Верификация контракта: Etherscan API **V2** — ключ брать на **etherscan.io** (единый ключ работает и для Basescan), endpoint `https://api.etherscan.io/v2/api?chainid=84532`. ## Финальный UX-план (реализация — заменяет открытый вопрос «Ревизия UX»)
Цель: довести до «готов к выпуску» под продуктового iGaming-оценщика. Дизайн-цель — Limbo-канон Stake (референс `screen/stake-limbo.png`, локальный/gitignored) + наша проверяемая честность как преимущество над Stake. Всё — на фронте, БЕЗ редеплоя контракта.

**P0 — модель счёта, вход, онбординг (ядро ревизии):**
- Один баланс **«Счёт казино»** на виду (слить встроенный кошелёк + `balances[]` автодепозитом). Спрятать слова smart account / AA / EOA / «встроенный кошелёк».
- Вход — **только email** (`inAppWallet({ auth: { options: ["email"] } })`), gasless. Убрать Google/Apple/Facebook/phone/passkey/MetaMask как способы входа (ловушка двух кошельков).
- **Игра видна сразу** — убрать стену «подключи кошелёк», всегда рендерить игру + дашборд. Прогрессивный вход: одна кнопка **«Войти и сыграть»** → email-вход → авто кран+депозит (молча, gasless) → ставка; вошёл → «Сыграть»; занят → «Ждём результат…».
- Множитель в покое = **«1.00×»** (не «—»), на ставку — прокрут вверх и замирание на результате.
- **Вывод:** «Вывести на свой кошелёк» = поле вставки внешнего адреса → отправка (gasless). Единственный/легитимный вывод наружу; существующую внешнюю отправку переименовать, убрать ощущение «третьего счёта». Обычный выигрыш остаётся на «Счёте казино».

**P1 — экран игры (Stake-канон) + дифференциатор:**
- Двухколоночно: панель ставки (слева) + крупный центральный множитель.
- Панель: сумма ставки + кнопки **½ / 2×**, целевой множитель, **Win Chance %**, выплата, primary **«Сыграть»**.
- Тексты по-русски: «Сыграть» вместо `PLACE BET`; понятная строка честности вместо `provably fair · Pyth Entropy`; на первом входе — мягкая подсказка вместо пугающего красного «нет баланса».
- **Лента последних ставок** из on-chain `BetSettled`, каждая строка → Basescan (деко + транспарентность, обход Stake).
- Дашборд самоаудита (edge/hold/RTP — язык юнит-экономики) + верификатор честности на витрине. Нарратив «прозрачнее Stake» (Stake критикуют за невидимый RNG-аудит).

**P2 — полировка:** мобильный адаптив (базовый), состояния/ошибки, бренд.
